# wiki-curator

## Что делает

Замыкает петлю **Self-Update** из методологии LLM Wiki (см. `wiki/topics/markdown-knowledge-base.md`):

```
Сессия → задача → wikI-loader загрузил → решил → ?
                                                  ↓
                                       wiki-curator решает:
                                       появилась ли новая инфа?
                                                  ↓
                                       ДА → классифицировать → commit
                                       НЕТ → ничего
```

## Когда срабатывает (триггеры)

**Срабатывает** в конце значимой задачи, если был хотя бы один из:
- 🚨 Новая ошибка, которая повторялась 2+ раза
- 🚨 Новая среда/сервис (например, добавил OpenAI как fallback к YandexGPT)
- 🚨 Новый эффективный паттерн (например, «Webhook node оказался лучше Telegram Trigger в кейсе X»)
- 🚨 Изменение интерфейса существующего сервиса (Telegram API, YandexGPT API)
- 🚨 Конец сессии (обновить `wiki/session-summaries.md`)

**НЕ срабатывает** после:
- ❌ Решения task-specific проблемы (исправил опечатку в README этого проекта)
- ❌ Одноразового хака (поменял IP VPS)
- ❌ Простого ответа на вопрос
- ❌ Переписывания существующего файла без нового знания

## Алгоритм

### Шаг 1. Self-eval (рассуждение вслух)

Перед тем как что-то менять, пройди дерево:

```
Q1. Появилась ли в этом запросе НОВАЯ ИНФОРМАЦИЯ?
    (т.е. то, чего нет в wiki/ENVIRONMENT/MISTAKES)
    ├─ НЕТ → обновление не нужно. Сообщи «wiki не требует обновления».
    └─ ДА  → Q2

Q2. Это общий урок / паттерн / новая среда?
    (применим в будущих проектах, не только этот раз)
    ├─ НЕТ → Q3 (только project-specific)
    └─ ДА  → обновление НУЖНО → Q4

Q3. Это полезно для будущих сессий?
    ├─ ДА  → обновление НУЖНО
    └─ НЕТ → не обновлять (но записать в локальный scratchpad / memory)

Q4. Это структурное изменение wiki (новый файл, новая папка)?
    ├─ ДА  → требует одобрения пользователя (не auto-commit)
    └─ НЕТ → auto-commit по протоколу ниже

Q5. Уже есть запись про это в wiki?
    ├─ ДА  → дополнить, не дублировать
    └─ НЕТ → создать новую запись
```

### Шаг 2. Классифицировать, что обновлять

| Тип инфы | Куда | Формат |
|----------|------|--------|
| Новая ошибка с root cause | `MISTAKES.md` (новая секция `### X.Y`) | `### X.Y ❌ [название] / Когда / Симптом / Причина / Решение` |
| Изменение/добавление сервиса | `ENVIRONMENT.md` (новая подсекция) | по схеме существующих секций |
| Новый эффективный паттерн | `wiki/COUNCIL_RECOMMENDATIONS.md` или новый `wiki/topics/<name>.md` | с примером и контр-примером |
| Устаревшее | пометить `status: superseded` в frontmatter | не удалять, индексировать |
| Открытый вопрос на будущее | `wiki/pending.md` | `- [ ] вопрос (из сессии YYYY-MM-DD)` |
| Конец сессии | `wiki/session-summaries.md` | новая секция «Сессия N» |
| Любое изменение | `wiki/log.md` (changelog) | строка в стиле `## YYYY-MM-DD` |

### Шаг 3. Проверить capabilities

Узнать режим записи (смотри `memory topic llm-manifest-state`):

| Режим | Условие | Действие |
|-------|---------|----------|
| **CAN-WRITE-FULL** | есть `GITHUB_LLM_MANIFEST_TOKEN` (PAT) | коммитить напрямую |
| **CAN-READ-ONLY** | нет токена | подготовить diff в формате `git apply` |
| **DEGRADED** | репо недоступно | вывести полный текст изменённых файлов |
| **OFFLINE** | нет сети | то же, что DEGRADED + предложить пользователю сделать позже |

### Шаг 4A. CAN-WRITE-FULL — коммит напрямую

```bash
cd /workspace/llm_manifest

# 1. Проверить чистоту
git status

# 2. Показать diff пользователю (dry-run)
git diff --stat

# 3. Если пользователь одобрил — коммитить
git add <files>
git commit -m "wiki: [section] [brief description]

Подробнее: ..."

# 4. Push
git push origin main
```

**Сообщение коммита** — в формате:
- `MISTAKES.md: добавлен §X.Y про [короткое название]`
- `ENVIRONMENT.md: обновлён §X.Y про [сервис]`
- `wiki/topics: новый файл <name>.md`
- `wiki/log.md: запись о сессии N`

**Что НЕ коммитить автоматически:**
- `wiki/SCHEMA.md`, `wiki/AGENTS.md` — структурные изменения, требуют ручного одобрения
- `wiki/raw/*` — только ручное добавление
- Файлы с `status: locked` в frontmatter

### Шаг 4B. CAN-READ-ONLY — подготовить diff

```bash
cd /workspace/llm_manifest

# Сохранить diff
git diff > /tmp/wiki-update-$(date +%Y%m%d-%H%M).patch

# Показать пользователю
cat /tmp/wiki-update-*.patch
```

**Показать пользователю шаблон:**

```markdown
## 🔄 Обновление wiki после этой задачи

**Категория:** [ошибка | среда | паттерн | ...]
**Файлы:** `MISTAKES.md`, `wiki/log.md`
**Применить:**

```bash
cd /path/to/llm_manifest
git apply /tmp/wiki-update-YYYYMMDD-HHMM.patch
git add -A
git commit -m "wiki: [section] [brief]"
git push
```

**Патч:**

\`\`\`diff
--- a/MISTAKES.md
+++ b/MISTAKES.md
@@ -... @@
### X.Y ❌ [название]
...
\`\`\`

**Подтверди, что применил — я проверю в следующей сессии.**
```

### Шаг 4C. DEGRADED / OFFLINE — полный текст

Вывести пользователю **полный текст** каждого изменённого файла, чтобы он скопировал руками. Длинно, но единственный способ.

### Шаг 5. Обновить runtime state

После успешного коммита / patch — обновить memory topic `llm-manifest-state`:
- новый SHA
- пометка «обновлено X.Y: <короткое описание>»

## Запрещено

- ❌ Коммитить без явного одобрения пользователя, если изменение затрагивает `SCHEMA.md` или `AGENTS.md`
- ❌ Перезаписывать существующие секции без сохранения версии (добавлять `Updated: YYYY-MM-DD` в frontmatter)
- ❌ Дублировать записи (если урок уже есть — дополнить, не плодить копии)
- ❌ Писать секреты в wiki (даже в `.env.example` — `MISTAKES.md §14.8`)
- ❌ Пытаться fetch github.com для проверки, если режим DEGRADED (только `git status` / `git log` локально)

## Связь с другими skill'ами

- **`wiki-loader`** — загрузил контекст, теперь нужно его обновить. Этот skill.
- **`gstack-team`** — Phase 7 (ship) вызывает `wiki-curator` в конце, чтобы обновить базу знаний по итогам спринта.
- **memory topic `llm-manifest-state`** — здесь хранится режим записи и SHA, обновляется в конце.

## Формат ответа пользователю после коммита

```markdown
✓ wiki обновлена: [файл:секция]
  Commit: <sha>
  Log: [что изменилось в 1 предложении]

Если есть ещё — продолжаем. Если нет — сессия закрыта.
```

## Smoke test

Чтобы проверить, что skill сработал, в ответе должно быть:
- ✅ Дерево решений Q1-Q5 пройдено
- ✅ Классификация (куда писать — MISTAKES / ENVIRONMENT / topics)
- ✅ Соблюдён режим записи (CAN-WRITE / DEGRADED)
- ✅ Commit SHA или diff показан пользователю
- ✅ `wiki/log.md` обновлён в любом случае
