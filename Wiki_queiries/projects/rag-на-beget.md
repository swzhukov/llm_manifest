## Цель

Построить **RAG-систему** (Retrieval-Augmented Generation) на VPS от Beget. Для личного использования и/или для команды Первого Бита.

---

## Что такое RAG (кратко)

RAG = поиск релевантных документов в базе + генерация ответа LLM с учётом найденного. **Альтернатива** LLM Wiki (методология Карпатого) и **дополнение** к ней.

**Где применяется:**
- Поиск по корпоративной базе знаний.
- Чат-бот поддержки.
- Анализ документов.
- Персональный помощник (поиск по своим файлам).

---

## Требования к железу (от LLM)

| Сценарий | vCPU | RAM | Хранилище |
|----------|------|-----|-----------|
| **RAG с LLM через API** (минимум) | 2-4 | 4-8 ГБ | 30-50 ГБ SSD |
| **RAG с локальной LLM** (7B-13B) | 4-8 | 16-32 ГБ | 200+ ГБ NVMe |
| **Production RAG** | 8-16 vCPU | 16-32 ГБ+ | 200+ ГБ NVMe |

**Важно:** Beget не предлагает GPU-серверы → **локальные LLM только через CPU**, что медленно. Рекомендуется использовать **API моделей** (YandexGPT, OpenAI).

## Готовые OpenSource-решения

LLM рекомендует:
- **Dify** — платформа для LLM-приложений, есть RAG.
- **AnythingLLM** — приватный чат с документами.
- **RAGFlow** — open-source RAG от InfiniFlow.

Все работают через `docker compose`.

## Оценка Mavis

**Состояние:** 20% готово. Есть архитектурный план, нет реализации.

**Главный риск:** Сергей **не администратор Linux**. Самостоятельная настройка Beget VPS + Docker + Postgres + Traefik + RAG — это **много слоёв** для одного человека.

**Что я бы рекомендовал (если бы Сергей спросил):**
- **Начать с managed-сервиса** (Dify Cloud, AnythingLLM Cloud) — для быстрого теста.
- **Мигрировать на Beget** только когда станет ясно, что RAG нужен в проде.
- **Использовать YandexGPT** для embeddings (российская LLM, проще юридически).
- **Не делать локальную LLM** на Beget (медленно без GPU).
