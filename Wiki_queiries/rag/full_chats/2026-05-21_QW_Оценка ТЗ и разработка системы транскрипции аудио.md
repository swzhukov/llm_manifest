## USER

Техническое задание: Система пакетной транскрипции аудио с облачным сохранением прогресса
1. Контекст и цель продукта
У пользователя есть множество аудиофайлов (лекции, интервью, созвоны), которые необходимо превратить в текст с помощью стороннего API распознавания речи. Файлы могут быть большого размера (несколько часов), а сеанс обработки может прерваться из-за нестабильного интернета или перезагрузки среды выполнения (Google Colab).
Цель — создать инструмент, который:

Автоматически загружает аудио, подготавливает его, отправляет на транскрипцию и собирает результат.

Устойчив к перезапускам: прогресс и результаты сохраняются в облачное хранилище (Mail.ru WebDAV), при повторном запуске необработанные файлы докачиваются, а дубликаты пропускаются.

Не требует ручного отслеживания состояния и минимизирует повторную работу.

2. Ключевая ценность
Экономия времени: пакетная обработка десятков файлов с автоматическим возобновлением.

Надёжность: даже после сбоя не теряются уже распознанные фрагменты.

Контроль дубликатов: файлы проверяются по MD5‑хешу, исключая повторную транскрипцию одного и того же контента.

3. Основные функциональные требования
3.1. Загрузка и валидация аудио
Пользователь может загрузить один или несколько файлов через стандартный диалог (в среде Colab) или указать путь к папке с аудио (для локального/серверного режима).

Поддерживаемые форматы: MP3, WAV, M4A, FLAC, OGG, AAC, WMA.

Каждый файл проверяется на читаемость, ненулевую длительность и формат.

Уже обработанные файлы (по хешу) помечаются как пропущенные с сообщением пользователю.

3.2. Предобработка аудио
Все файлы приводятся к оптимальному формату для API: моно, 16 кГц, WAV.

Если исходный файл уже соответствует требованиям, перекодирование не выполняется.

Аудио длительностью более 1 часа разбивается на чанки по 1 часу с сохранением порядка.

3.3. Взаимодействие с API транскрипции
Используется заданный внешний сервис (ASR_URL), аутентификация по статическому токену.

Для каждого чанка:

Отправка файла на эндпоинт /start_transcribing с параметрами диаризации (если нужно).

Периодический опрос статуса /get_status (интервал и количество попыток настраиваются).

При достижении статуса «done» или прогресса 100% — получение текста через /get_file.

Результат накапливается, собирается в единый текст с заголовками чанков.

3.4. Управление прогрессом и сохранение состояния
В облачном хранилище хранится файл progress.json, в котором зафиксированы:

Какие файлы начаты, сколько чанков обработано.

Для каждого чанка — статус (завершён / не начат) и полученный текст.

При запуске система загружает progress.json и определяет список файлов, требующих доработки.

Пользователь может выбрать: продолжить незавершённые файлы или начать обработку новой партии.

3.5. Контроль дубликатов файлов
Ведётся отдельная база хешей (file_hashes.json) в облаке.

Перед началом обработки для каждого файла вычисляется MD5-хеш. Если хеш совпадает с ранее обработанным файлом (даже под другим именем), транскрипция пропускается, а готовый текст загружается из облака.

Хеш‑база синхронизируется с облаком при каждом обновлении.

3.6. Сохранение и выдача результатов
Итоговый текст для файла сохраняется в локальную папку results/ и параллельно загружается в облако.

По окончании всей сессии формируется ZIP‑архив со всеми текстами.

Пользователю предлагается скачать архив.

3.7. Логирование
Ход выполнения пишется в лог‑файл и выводится в консоль.

Для длительных операций отображается прогресс‑бар (tqdm) или периодические сообщения с процентами.

4. Нефункциональные требования
4.1. Надёжность и отказоустойчивость
Все сетевые операции (API, WebDAV) должны содержать повторные попытки с экспоненциальной задержкой при ошибках 5xx, 429 и таймаутах.

После любого аварийного завершения временные файлы и чанки должны очищаться (механизм finally или контекстные менеджеры).

При отсутствии связи с облаком система не должна падать — ошибка логируется, но локальный прогресс сохраняется для последующей синхронизации (если возможно).

4.2. Безопасность
Токен API и пароль облака не хранятся в коде. Они загружаются из переменных окружения или из безопасного хранилища учётных данных Colab (userdata).

Все соединения с облаком идут по HTTPS, но без отключения проверки сертификатов (в production).

4.3. Кроссплатформенность и переносимость
Код должен работать как в Google Colab, так и на локальной машине/сервере. Различия (загрузка файлов через colab.files) инкапсулируются за абстрактным интерфейсом «источник аудио».

Зависимости фиксируются в requirements.txt.

4.4. Производительность
Вычисление MD5 файла не должно загружать весь файл в память (читать чанками).

При конвертации аудио используются временные файлы, а не хранение в оперативной памяти больших объектов.

5. Архитектура (рекомендуемая)
Система строится по модульному принципу. Основные компоненты:

Config — централизованные настройки (пути, тайминги, параметры API).

AudioProcessor — валидация, конвертация, разбиение на чанки.

APIClient — инкапсуляция HTTP-запросов к сервису распознавания, обработка ответов, ретраи.

CloudStorage — интерфейс для операций с WebDAV (upload/download, проверка существования, создание папок).

ProgressManager — работа с progress.json: загрузка, сохранение, обновление прогресса.

HashManager — ведение словаря «имя/путь: хеш», сохранение в облако, поиск дубликатов.

TranscriptionOrchestrator — главный оркестратор, который принимает список файлов и координирует их обработку, используя вышеперечисленные сервисы.

CLI / Colab UI — точка входа, запрос файлов, вывод результатов.

6. Ключевые сценарии работы
Сценарий 1: Первичный запуск с новыми файлами
Пользователь загружает 3 аудиофайла.

Система аутентифицируется в облаке, синхронизирует progress.json и file_hashes.json.

Для каждого файла вычисляется хеш, проверяется по базе. Все три — новые.

Файлы конвертируются, разбиваются на чанки при необходимости.

Чанки отправляются в API; по мере готовности текст сохраняется в прогресс и облачную папку results/.

После завершения всех файлов создаётся архив, предлагается скачать.

Сценарий 2: Перезапуск после обрыва
Пользователь перезапускает среду выполнения.

Система обнаруживает в progress.json незавершённые файлы.

Пользователю предлагается продолжить обработку. При согласии обрабатываются только недостающие чанки.

Ранее полученные тексты для завершённых файлов не запрашиваются повторно.

После доработки формируется финальный архив.

Сценарий 3: Дубликат файла
Пользователь загружает файл, идентичный ранее обработанному (тот же MD5).

Система находит его в хеш‑базе, скачивает готовый результат из облака, уведомляет: «Файл уже обработан, результат восстановлен».

Транскрипция не запускается.

8. Критерии приёмки
При запуске на тестовом наборе из 5 файлов разной длительности и форматов все файлы успешно транскрибируются.

Искусственный обрыв соединения (отключение сети на этапе транскрипции) и повторный запуск приводят к успешному завершению без потери данных.

## 1. Оценка адекватности ТЗ и предложения по улучшению

**Общая оценка:** ТЗ — **детальное, продуманное и технически грамотное**. Архитектура модульная, требования к надёжности (ретраи, чанкирование, облачное сохранение прогресса) — сильная сторона. Однако есть несколько моментов, которые стоит улучшить или пересмотреть.

### Что можно улучшить:

| Аспект | Проблема | Предлагаемое решение |
|---|---|---|
| **API транскрипции** | В ТЗ описан низкоуровневый API (/start_transcribing, /get_status, /get_file). | Использовать **newton-cli** как единую точку входа. Он сам управляет задачами, опрашивает статус и возвращает результат. |

### Альтернативное решение (более простое)

Если цель — просто транскрибировать файлы с диаризацией и сохранять результат в облако, можно обойтись **скриптом-обёрткой над newton-cli** без сложного оркестратора. Однако при большом количестве файлов и нестабильном окружении **пакетный подход с отслеживанием прогресса оправдан**.

## 2. Архитектура системы

Система построена по модульному принципу, как рекомендовано в ТЗ. Основные компоненты:

```
transcription_system/
├── config.py            # Конфигурация (пути, URL, таймауты)
├── audio_processor.py   # Валидация, конвертация, разбиение на чанки
├── api_client.py        # Взаимодействие с newton-cli (subprocess)
├── cloud_storage.py     # WebDAV-клиент (webdavclient3)
├── progress_manager.py  # Управление progress.json
├── hash_manager.py      # Управление file_hashes.json
├── orchestrator.py      # Главный оркестратор
├── cli.py               # Точка входа (CLI + Colab)
├── requirements.txt
├── .env.example
└── README.md
```

### Поток данных:

1. Пользователь запускает `cli.py` (в Colab или локально).
2. `Orchestrator` загружает список файлов.
3. Для каждого файла:
   - Вычисляется MD5-хеш, проверяется в `HashManager`.
   - Если файл новый: аудио валидируется, при необходимости конвертируется в WAV 16kHz mono, разбивается на чанки.
   - Чанки отправляются в `APIClient`, который вызывает `newton transcribe` с движком `diarize`.
   - Прогресс и результаты сохраняются в `ProgressManager` и выгружаются в облако.
   - Если файл уже обработан (дубликат), результат скачивается из облака.
4. По окончании создаётся ZIP-архив, предлагается скачать.

### Обработка ошибок и возобновление:

- При запуске `ProgressManager` загружает `progress.json` из облака.
- Если есть незавершённые файлы, пользователю предлагается продолжить.
- Все сетевые операции обёрнуты в ретраи с экспоненциальной задержкой.
- Временные файлы удаляются в `finally`.

### 3.2. Аудиопроцессор

```python
# audio_processor.py
import hashlib
import os
import tempfile
import logging
from pydub import AudioSegment
from pathlib import Path
from config import TARGET_SAMPLE_RATE, TARGET_CHANNELS, CHUNK_DURATION_SEC

logger = logging.getLogger(__name__)

def compute_md5(file_path: str) -> str:
    """Вычисляет MD5-хеш файла чанками."""
    hash_md5 = hashlib.md5()
    with open(file_path, "rb") as f:
        for chunk in iter(lambda: f.read(8192), b""):
            hash_md5.update(chunk)
    return hash_md5.hexdigest()

def validate_audio(file_path: str) -> bool:
    """Проверяет, что файл читается и имеет ненулевую длительность."""
    try:
        audio = AudioSegment.from_file(file_path)
        return len(audio) > 0
    except Exception:
        return False

def convert_to_wav_mono_16k(input_path: str, output_path: str) -> None:
    """Конвертирует аудио в WAV, 16 кГц, моно."""
    audio = AudioSegment.from_file(input_path)
    if audio.channels > TARGET_CHANNELS:
        audio = audio.set_channels(TARGET_CHANNELS)
    if audio.frame_rate != TARGET_SAMPLE_RATE:
        audio = audio.set_frame_rate(TARGET_SAMPLE_RATE)
    audio.export(output_path, format="wav")

def split_into_chunks(file_path: str, chunk_duration_sec: int = CHUNK_DURATION_SEC) -> list[str]:
    """Разбивает аудио на чанки и возвращает список путей."""
    audio = AudioSegment.from_file(file_path)
    total_sec = len(audio) / 1000.0
    if total_sec <= chunk_duration_sec:
        return [file_path]

    chunk_paths = []
    for i, start_ms in enumerate(range(0, len(audio), chunk_duration_sec * 1000)):
        chunk = audio[start_ms:start_ms + chunk_duration_sec * 1000]
        fd, chunk_path = tempfile.mkstemp(suffix=".wav", prefix=f"chunk_{i}_")
        os.close(fd)
        chunk.export(chunk_path, format="wav")
        chunk_paths.append(chunk_path)
    return chunk_paths
```

### 3.3. API-клиент (newton-cli)

```python
# api_client.py
import subprocess
import json
import logging
import shlex
from config import NEWTON_TOKEN

logger = logging.getLogger(__name__)

def transcribe_with_diarization(audio_path: str) -> dict:
    """
    Запускает newton-cli с движком diarize.
    Возвращает словарь с результатом (JSON).
    """
    cmd = f"newton transcribe {shlex.quote(audio_path)} --engine diarize"
    logger.info(f"Running: {cmd}")
    try:
        result = subprocess.run(
            cmd,
            shell=True,
            capture_output=True,
            text=True,
            env={"NEWTON_TOKEN": NEWTON_TOKEN, **__import__('os').environ},
            check=True
        )
        # newton diarize возвращает JSON в stdout
        output = json.loads(result.stdout)
        return output
    except subprocess.CalledProcessError as e:
        logger.error(f"newton-cli failed: {e.stderr}")
        raise
    except json.JSONDecodeError as e:
        logger.error(f"Failed to parse JSON output: {e}")
        raise
```

```python
# cloud_storage.py
import logging
from webdav3.client import Client
from config import WEBDAV_HOSTNAME, WEBDAV_LOGIN, WEBDAV_PASSWORD

logger = logging.getLogger(__name__)

options = {
    'webdav_hostname': WEBDAV_HOSTNAME,
    'webdav_login': WEBDAV_LOGIN,
    'webdav_password': WEBDAV_PASSWORD
}
client = Client(options)

def upload_file(local_path: str, remote_path: str) -> None:
    """Загружает файл в облако."""
    logger.info(f"Uploading {local_path} -> {remote_path}")
    client.upload_sync(remote_path=remote_path, local_path=local_path)

def download_file(remote_path: str, local_path: str) -> None:
    """Скачивает файл из облака."""
    logger.info(f"Downloading {remote_path} -> {local_path}")
    client.download_sync(remote_path=remote_path, local_path=local_path)

def file_exists(remote_path: str) -> bool:
    """Проверяет существование файла в облаке."""
    return client.check(remote_path)
```
