```
  Выручка = 100% от постпроектных услуг (SLA-контракты)

КОМИССИЯ КОММЕРЧЕСКОЙ СЛУЖБЫ:
```

## 🔍 框架 АНАЛИЗА: ЧТО ИСКАТЬ

### УРОВЕНЬ 1: ФИНАНСОВАЯ ЗДОРОВЬЕ ЦФО

**Главный вопрос:** Каждый ЦФО прибылен? Или есть скрытые убытки?

**Что анализировать:**
1. **Маржинальность по ЦФО** (доход / расходы)
   - Проверить, кто платит ВЗР и сколько
   - Расчитать реальную маржу каждого ЦФО
   
2. **ВЗР-дефицит как финансовая проблема**
   - Если Центр разработки не может поставить ВЗР → это убыток для покупающих ЦФО
   - Расчитать стоимость этого дефицита (часы × ставка программиста)
   
3. **Встреч как проблема маржи СЛЭЙ**
   - Совещания = low-margin консультации
   - Расчитать упущенный доход от встреч
   - Сравнить доход встреч vs проектной работы

### УРОВЕНЬ 2: ОПЕРАЦИОННАЯ ЭФФЕКТИВНОСТЬ

**Главный вопрос:** Используются ли ресурсы оптимально?

**Что анализировать:**
1. **Utilization rate по типам сотрудников**
   - Программисты: должны быть 85-95% на проектах (не администрация)
   - ПМы: должны быть 70-80% на проектах (20-30% управление)
   - Аналитики: должны быть 60-70% на проектах (30-40% RFI/консультации)

2. **Фрагментация работ**
   - Мелкие задачи < 0.5ч = потеря на переключение контекста
   - Расчитать стоимость фрагментации (коэффициент ~1.3x переработки)

3. **Зависимости между проектами**
   - Какие проекты "доноры" ресурсов?
   - Какие проекты "потребители"?
   - Есть ли bottleneck-проекты, которые тормозят остальное?

### УРОВЕНЬ 3: СКРЫТЫЕ ФИНАНСОВЫЕ РИСКИ (НЕОЧЕВИДНЫЕ ИНСАЙТЫ)

**Главный вопрос:** Какие проблемы "бомбят" прибыльность, но никто их не видит?

**Что искать:**
1. **Синдром "ключевого человека"**
   - Есть ли сотрудники, от которых зависит 20%+ прибыли?
   - Если уходит → компания теряет X млн рублей
   - Пример: Кучухидзе на проекте "Русские корни" (46% часов)

2. **Проекты-убытки**
   - Проекты с очень высокой долей встреч (>50%)
   - Проекты с множеством мелких задач (средний размер <1ч)
   - Проекты, где один человек работает 100% (нет резервирования)
   - Проекты с очень низким ROI времени

3. **ВЗР-волатильность и его скрытая стоимость**
   - Если ВЗР дефицитны → другие ЦФО платят больше за фрилансеров/контрактников
   - Это может быть заметно как "необычные расходы"
   - Калькулировать реальную стоимость дефицита

4. **Профессиональное выгорание как финансовый риск**
   - Сотрудники с паттернами: >10ч в день + >70% дней, падение эффективности к концу месяца
   - Риск: Могут уйти (потеря компетенций) или упадет качество (клиентские жалобы, переделки)

5. **Некачественная отчетность как скрытая убыток**
   - Копипаст описания = невозможно аналитировать прибыльность
   - Это мешает расчитать реальный ROI проектов и корректировать цены
   - Потери: 5-10% от дохода на переговорах и неправильном ценообразовании

6. **Асимметрия нагрузки между ЦФО**
   - УКП 5.1 может быть перегруженным, а Отдел отраслевого развития недоутилизирован
   - Или наоборот: Отдел растет быстро, отбирает ресурсы, и УКП 5.1 становится убыточным
   - Это может быть стратегическая ошибка

## 📊 СТРУКТУРА АНАЛИЗА И ПЛАН ИССЛЕДОВАНИЯ

### ФАЗА 1: ЗАГРУЗКА И КЛАССИФИКАЦИЯ ДАННЫХ

**Входные данные:** Excel/CSV с колонками: День, ФИО, Тип работы, Описание, Проект, Часы

**Шаги:**
1. Загрузить данные, очистить от пустот
2. **Классифицировать каждую строку по ЦФО:**
   - По названию проекта → определить ЦФО (УКП 5.1, СЛЭЙ, и т.д.)
   - Если не ясно → попросить уточнить
3. **Классифицировать по типам работ:**
   - Проектная (коммерческая)
   - Внутренняя (разработка компетенций, архитектура)
   - Администрирование (отчетность, планирование)
   - Совещания (встречи, планерки, синхросы)
   - Обучение
4. **Определить роли сотрудников:**
   - Программисты (высокая доля проектной работы >80%)
   - ПМы (много совещаний, но есть проектная работа)
   - Аналитики (балансируют между проектом и консультациями)
   - Коммуникаторы (много встреч, мало проектной работы)

### ФАЗА 2: ФИНАНСОВЫЙ АНАЛИЗ ПО ЦФО

**Для каждого ЦФО рассчитать:**

```
1. СТРУКТУРА ТРУДОЗАТРАТ:
   - Всего часов
   - Проектные часы (коммерческие)
   - Внутренние часы
   - Совещания
   - Админ
   
2. МАРЖИНАЛЬНОСТЬ (ориентировочно):
   - Доход ~ Проектные часы × средняя ставка ЦФО
   - Расходы ~ Всего часов × средняя зарплата сотрудника
   - Маржа = (Доход - Расходы) / Доход × 100%
   
3. ВЗР-МЕТРИКИ (если ЦФО платит ВЗР):
   - Сколько часов в ВЗР?
   - Нормально ли это или дефицит?
   - Какая стоимость дефицита?
   
4. СОВЕЩАНИЕ-МЕТРИКА (% встреч от времени):
   - <15% = здорово (встречи не мешают)
   - 15-30% = нормально (управляемо)
   - 30-50% = проблема (маржа подавлена)
   - >50% = критичная проблема (низкая маржа)
```

### ФАЗА 3: ПЕРСОНАЛЬНЫЙ АНАЛИЗ С АКЦЕНТОМ НА ФИНАНСОВЫЕ РИСКИ

**Для каждого сотрудника рассчитать:**

```
1. ВКЛАД В ЦФО:
   - Часов в своей ЦФО
   - Доля от выручки ЦФО (если это программист/аналитик)
   - Риск "ключевого человека" (если он >15% выручки)
   
2. ФИНАНСОВЫЙ PROFILE:
   - Проектные часы (высокие = хороший ROI)
   - Встречи (высокие = низкий ROI)
   - Администрирование (должно быть <10%)
   
3. РИСК ВЫГОРАНИЯ (финансовый угол):
   - Если выгорит → сколько мы потеряем?
   - Критичные люди должны быть приоритетом
   
4. КАЧЕСТВО ОТЧЕТНОСТИ:
   - Уникальность описаний (% уникальных записей)
   - Это влияет на видимость затрат и маржинальность
```

### ФАЗА 4: СКРЫТЫЕ ПРОБЛЕМЫ И ИНСАЙТЫ

**Ищем:**

1. **Проекты-убытки**
   ```
   Признаки:
   - >50% совещаний
   - Средний размер задачи <0.5ч (фрагментация)
   - 1-2 человека вместо команды (нет резервирования)
   - Много копипаста в описаниях (low transparency)
   ```

2. **ВЗР-дефицит как скрытая бомба**
   ```
   Признаки:
   - Топ-программисты работают >210 часов (переработка на 25%)
   - Среднее значение для "ресурсного центра" низкое
   - ЦФО покупающие ВЗР платят комиссию сверху
   Финансовый эффект: 3-5 млн рублей убытка в месяц
   ```

3. **Встречи как маржинальная проблема**
   ```
   Признаки:
   - СЛЭЙ: >30% встреч
   - Постпроектные услуги имеют низкую маржу
   Финансовый эффект: 2-4 млн рублей упущенного дохода в месяц
   ```

4. **Асимметрия загруженности ЦФО**
   ```
   Признаки:
   - Один ЦФО работает 100%, другой <50%
   - Ресурсы отбираются у одного в пользу другого
   Финансовый эффект: Убыток менее загруженного ЦФО
   ```

### ФАЗА 5: ВЫВЕДЕНИЕ РЕКОМЕНДАЦИЙ

**Для каждой проблемы:**

```
Формат:
- ПРОБЛЕМА: Одна фраза (что именно ломает прибыль)
- СИМПТОМЫ: Из данных октября (конкретные цифры)
- ФИНАНСОВЫЙ ЭФФЕКТ: Сколько денег теряем (в млн рублей/месяц)
- РЕШЕНИЕ: Конкретное действие (не "улучшить" а как именно)
- KPI: Как будем измерять успех
```

                return 'Другое'
        
        профили_сотрудников['Роль'] = профили_сотрудников.apply(определить_роль, axis=1)
        self.данные = self.данные.merge(профили_сотрудников[['Роль']], 
                                       left_on='Пользователь', right_index=True)
        print("✅ Определение ролей завершено")

        """Рассчитывает ключевые финансовые и операционные метрики"""
        
        # Метрики по ЦФО
        метрики_цфо = self.данные.groupby('ЦФО').agg({
            'Рабочих часов': 'sum',
            'Пользователь': 'nunique',
            'Проект': 'nunique'
        }).round(2)
        
        метрики_цфо.columns = ['ВсегоЧасов', 'УникальныеСотрудники', 'УникальныеПроекты']
        
        # Структура работ по ЦФО
        структура_работ = pd.crosstab(self.данные['ЦФО'], self.данные['ТипРаботы'], 
                                    values=self.данные['Рабочих часов'], 
                                    aggfunc='sum').fillna(0)
        
        # Метрики по сотрудникам (для выявления ключевых людей)
        метрики_сотрудников = self.данные.groupby(['Пользователь', 'ЦФО', 'Роль']).agg({
            'Рабочих часов': 'sum',
            'Проект': 'nunique',
            'ТипРаботы': lambda x: (x == 'Проектная').sum() / len(x) if len(x) > 0 else 0
        }).round(3)
        
        метрики_сотрудников.columns = ['ВсегоЧасов', 'КоличествоПроектов', 'Эффективность']
        метрики_сотрудников = метрики_сотрудников.sort_values('ВсегоЧасов', ascending=False)
        
        # Детектор переработки
        переработка = self.данные.groupby(['Пользователь', 'День']).agg({
            'Рабочих часов': 'sum'
        }).reset_index()
        
        переработка['Переработка'] = переработка['Рабочих часов'] > 8
        статистика_переработки = переработка.groupby('Пользователь').agg({
            'Переработка': 'mean',
            'Рабочих часов': 'mean'
        }).round(2)
        
        self.результаты = {
            'МетрикиЦФО': метрики_цфо,
            'СтруктураРабот': структура_работ,
            'МетрикиСотрудников': метрики_сотрудников,
            'СтатистикаПереработки': статистика_переработки,
            'ТопПроекты': self.данные.groupby('Проект')['Рабочих часов'].sum().sort_values(ascending=False).head(20),
            'ТопСотрудники': метрики_сотрудников.head(15)
        }
        
        print("✅ Расчет метрик завершен")
