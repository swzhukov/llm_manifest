### 🔧 Рекомендации по реализации полного отчета
Для полноценного отчета с подсветкой ошибок потребуется более глубокая разработка. Пользователи ПДО смогут запускать его как обычный отчет в разделе «Производство»[citation:2][citation:10].

1.  **Создайте новую обработку.** Добавьте ее в раздел «Производство». В качестве макета используйте табличный документ или систему компоновки данных (СКД).
2.  **Программируемая логика для проверки 2.3.2.** Проверку хронологии между этапами нужно реализовать на встроенном языке (например, в методе `ПриКомпоновкеРезультата`). Алгоритм:
    *   Сгруппируйте все этапы по заказу.
    *   Для каждого этапа найдите, для каких других этапов этого же заказа он выпускает полуфабрикаты (анализ состава выпуска).
    *   Сравните даты: если дата выпуска этапа-поставщика **позже** даты выпуска этапа-потребителя, это ошибка.
    *   Выделите все этапы, даты которых нарушают последовательность.
3.  **Визуальное оформление.** Настройте условное оформление в макете, чтобы строки, где любой из флагов ошибок (`ОшибкаДатыЗатрат`, `НетПередачиМатериалов` и т.д.) имеет значение **Истина**, выделялись красным цветом.

Если у вас есть доступ к конфигуратору и навыки разработки на 1С, этого плана будет достаточно для создания отчета. Если же таких возможностей нет, потребуется обратиться к специалисту по 1С.

    Заказ,

### 🔍 Как работает этот запрос

1.  **Источник данных** — табличная часть **«Выпуск»** документа «Этап производства». Именно здесь указывается, *для какого другого этапа* сделан выпуск (`Выпуски.ДляЭтапа`).
2.  **Связь** — запрос соединяет строку выпуска с документом-этапом, на который она ссылается (это и есть «этап-потребитель»).
3.  **Проверка** — сравниваются **ДатаПроизводства** текущего этапа и **ДатаПроизводства** этапа-потребителя. Если дата производителя **позже** — это нарушение (`НарушениеХронологии = Истина`).
4.  **Условия** — проверяются только связи в рамках **одного заказа** и для проведенных документов.

### 📌 Что этот запрос покажет

В результате вы получите список, где будут видны:
*   Этапы с нарушением хронологии (в начале списка).
*   Для каждого такого этапа — номер и дата этапа-потребителя, который был выпущен раньше.

**Пример из вашего аудио:** Если «Этап 11» (для «Этапа 13») выпущен 24-го числа, а «Этап 13» выпущен 17-го, то запрос отметит «Этап 11» как нарушитель.

### ⚠️ Важные ограничения и следующие шаги

1.  **Рекурсивные связи не проверяются** — запрос проверяет только прямые связи (когда этап А в выпуске прямо указан для этапа Б). Если цепочка длиннее (А → Б → В), то нарушение между А и В здесь не выявится. Для этого нужен более сложный алгоритм.
2.  **База для отчета** — этот запрос — **ядро для проверки по п.2.3.2**. Его логику нужно будет объединить с проверками из предыдущего ответа (2.3.1, 2.3.3, 2.3.4) в рамках одной **обработки или отчета**.
3.  **Внешние связи** — Если этап выпускает продукцию для этапа из **другого заказа**, такая связь в данном запросе игнорируется (условие `ДляЭтапа.ЗаказНаПроизводство = Выпуски.ЗаказНаПроизводство`). Если их нужно учитывать — это условие можно убрать.

Для создания итогового отчета, где строки с любой из ошибок будут подсвечиваться красным, потребуется разработать **обработку** на встроенном языке, которая объединит все проверки. Нужна помощь с алгоритмом для выявления сложных (рекурсивных) нарушений хронологии в цепочках этапов?

В итоговой выборке будут все этапы, которые для кого-то выпускают. Столбец **НарушениеХронологии** сразу укажет на проблемные. Пример из вашего аудио: если Этап 11 (поставщик для Этапа 13) выпущен 24 мая, а Этап 13 (потребитель) — 17 мая, то Этап 11 будет в списке с `НарушениеХронологии = Истина`.

### ⚠️ Ключевые моменты и ограничения:
- **Группировка через `MAX`**: Если в одном этапе несколько строк выпуска для **разных потребителей**, запрос покажет того потребителя, у которого максимальная дата выпуска. Это может скрыть частные нарушения. Для детального анализа нужно убрать группировку.

*   Если в табличной части `ВыходныеИзделия` поле ссылки на потребителя называется иначе (например, `ЭтапПотребитель`), измените `ВыходныеИзделия.ДляЭтапа` на соответствующее имя.
*   Для анализа за конкретный период раскомментируйте строку с условием по дате и определите параметры `&НачалоПериода` и `&КонецПериода`.

Запрос возвращает все связи между этапами, где указан потребитель, с пометкой о нарушении хронологии. Строки с `НарушениеХронологии = Истина` требуют внимания пользователя ПДО.
