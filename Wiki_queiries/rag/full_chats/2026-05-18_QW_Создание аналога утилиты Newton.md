## AI

Давай разберем, что из себя представляет Newton, и спроектируем его аналог. Ты прав, Newton — это, по сути, умная «прослойка» между пользователем и облачными API, что дает несколько важных преимуществ: не нужно думать об установке тяжелых ML-фреймворков, а вся бизнес-логика сводится к удобной работе с асинхронными задачами. Такой подход и правда выглядит очень практичным, и мы можем взять его за основу.

### 🧠 Анализ Newton: что под капотом?

Newton — это не монолитный AI-инструмент, а CLI-клиент, который делегирует всю «тяжелую» работу (распознавание речи, синтез, диаризацию) внешним серверам через API. Его ключевые особенности как раз и определяют архитектуру:

*   **API как бэкенд**: Все вычисления происходят на удаленных серверах, доступных по URL, которые можно переопределить через переменные окружения (`NEWTON_URL_V3`, `NEWTON_URL_TTS` и т.д.).[reference:0]
*   **Асинхронный режим**: Используется паттерн «отправил задачу — получил ID — опрашивай статус — забери результат».[reference:1][reference:2]
*   **Разнообразие «движков»**: Под капотом могут быть разные модели (NeMo, WhisperX, Fish-Speech), но для пользователя они выглядят как унифицированные команды или флаги (`--engine v3`, `--voice burunov`).[reference:3]
*   **Безопасность**: Авторизация по токену, который нужно получать и хранить в переменной окружения, а не хардкодить.[reference:4]

### 💡 Проектируем свой CLI: назовем его «Aether»

Раз мы решили делать аналог, давай назовем наш проект Aether. Его фундамент — это Python, а отталкиваться мы будем от архитектуры Newton, добавив современные библиотеки для надежности и удобства.

#### **1. Технологический стек (современный и удобный)**

Вот из чего будет состоять ядро Aether:

*   **Язык**: Python 3.10+ (как и у Newton, чтобы использовать все самые свежие фишки).[reference:5]
*   **Фреймворк для CLI**: [**Typer**](https://typer.tiangolo.com/). Это отличный выбор, потому что он использует родные аннотации типов Python, автоматически генерирует красивую справку и очень удобен для разработки.[reference:6][reference:7]
*   **HTTP-клиент**: [**HTTPX**](https://www.python-httpx.org/) с поддержкой `async/await`. Это сделает наш клиент быстрым и отзывчивым, особенно при опросе статуса задач. Для сравнения, в Python-клиенте RunPod также используется асинхронный подход для эффективного управления статусом задач.[reference:8]
*   **Управление конфигурацией**: [**Pydantic Settings**](https://docs.pydantic.dev/latest/concepts/pydantic-settings/). Позволит удобно загружать настройки из переменных окружения и валидировать их.
*   **Красивый вывод**: [**Rich**](https://rich.readthedocs.io/). Добавит интерактивные прогресс-бары при ожидании готовности задач и красиво оформит вывод команд (например, `voices` или `health`).

#### **2. Архитектура: обработка асинхронных задач**

Это сердце нашего инструмента. Когда ты просишь транскрибировать часовой файл, API не ответит мгновенно. Aether должен корректно обработать весь этот путь. Давай реализуем паттерн «Отправил-Запросил-Получил»:

1.  **Отправка (`POST /upload`)**: Загружаем аудиофайл на сервер и получаем в ответ `task_id`.
2.  **Ожидание (цикл опроса `GET /status/{task_id}`)**: Каждые 2-3 секунды проверяем статус задачи (`PENDING`, `PROCESSING`, `READY`, `ERROR`).[reference:9]
3.  **Получение результата (`GET /result/{task_id}`)**: Как только статус сменился на `READY`, скачиваем файл с текстом или аудио.

Эту логику можно оформить в виде переиспользуемой функции `async def poll_task(client, task_id)`.

#### **3. Структура CLI и его команды**

Мы полностью повторим удобную структуру Newton, но сделаем код более модульным.

##### **Основные команды**

*   `aether transcribe recording.wav`: Основная команда. Флаг `--engine` будет выбирать нужный «движок» (v3, whisper, diarize, stereo-v3), а `-o` — файл для сохранения.[reference:10]
*   `aether tts "Привет, мир!" -o hello.wav`: Синтез речи. Флаг `--voice` выберет голос, а `--format` задаст формат (wav, mp3, opus).[reference:11]
*   `aether voices`: Покажет список доступных голосов, разделяя их на системные и пользовательские.[reference:12]
*   `aether health`: Проверит статус всех настроенных серверов, аналогично `newton health`.[reference:13]

##### **Служебные команды**

*   `aether status <task_id>`: Покажет текущий статус задачи в читаемом или JSON-формате.
*   `aether result <task_id> -o output.txt`: Позволит скачать результат уже известной задачи, если ее ID сохранился с прошлого раза.[reference:14]

#### **4. Безопасность и конфигурация: никаких секретов в коде**

Это важнейший принцип, который подчеркивается и в документации Newton.[reference:15] Мы заложим его в фундамент Aether.

*   **Переменная окружения**: Все чувствительные данные (токен авторизации) будут читаться из переменной `AETHER_TOKEN`. В коде никогда не будет значений по умолчанию для токенов.
*   **Конфигурационный файл**: Для несекретных настроек создадим файл `~/.config/aether/config.yaml`. В нем можно будет переопределить URL-адреса сервисов, указать любимый язык транскрибации или голос по умолчанию.

```yaml
# ~/.config/aether/config.yaml
api:
  token_env: "AETHER_TOKEN" # Имя переменной с токеном
urls:
  transcribe_v3: "https://bit-asr3.1bitai.ru"
  tts: "https://bit-tts.1bitai.ru"
  # ...
defaults:
  engine: "v3"
  tts_voice: "burunov"
  language: "ru"
```

### 🚀 Стратегия развития: от простого к сложному

Такой дизайн позволяет двигаться поэтапно. Можно начать с малого и постепенно наращивать функциональность, не переписывая всё с нуля.

*   **Этап 1: MVP (Minimum Viable Product)**. Полностью повторить функциональность Newton как API-клиента, добавив современный UX (интерактивные прогресс-бары, цветной вывод).
*   **Этап 2: Гибридный режим**. Добавить поддержку локального Whisper для команды `transcribe`. Это повысит конфиденциальность и будет полезно, если нет интернета. Например, CLI-утилита `hark` полностью работает офлайн, используя локальную модель Whisper.[reference:16] Для TTS можно добавить локальный движок Piper. Утилита `voice` также поддерживает и GPU-ускорение через Kokoro, и работу на CPU через Piper.[reference:17]
*   **Этап 3: Self-Hosted**. Реализовать собственный бэкенд-сервер (например, на FastAPI), который будет принимать задачи от CLI. Это даст полный контроль над данными. Такой подход используют многие проекты, например, `psittsa` для TTS.[reference:18]

Как тебе такое направление? Мы можем начать с проектирования структуры данных для ответа от API диаризации или детально проработать архитектуру CLI-команды `transcribe`, чтобы ты мог сразу приступить к коду.
