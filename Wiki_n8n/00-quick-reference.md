# 00 — Quick Reference (шпаргалка)

Самая сжатая выжимка. Держи открытой во время работы.

## Нода: типы

| Тип | Вход | Выход | Форма | Назначение |
|---|---|---|---|---|
| **Trigger** | ❌ нет | ✅ да | закруглённые углы | старт workflow |
| **Action** | ✅ да | ✅ да | квадратная | действие / обработка |

> Из урока 6: «У триггера нет входа, только выход, и уголки закруглены. У экшена — вход и выход, углы прямые». Запомни — это главный визуальный маркер.

## 5 триггеров, которые покрывают 90% задач

| Триггер | Когда использовать |
|---|---|
| **Manual Trigger** | ручной запуск по кнопке (тестирование) |
| **n8n Form Trigger** | публичная форма-опросник, без своего фронта |
| **Email Trigger (IMAP)** | реакция на входящее письмо |
| **Local File Trigger** | изменения в папке на сервере |
| **Schedule Trigger** | по расписанию (cron) |
| **Telegram Trigger** | сообщение/нажатие в Telegram-боте |
| **Webhook** | внешний HTTP POST |

## 5 экшенов, которые покрывают 90% задач

- **HTTP Request** — вызов любого REST API
- **Code (JS/Python)** — произвольный код внутри ноды
- **Set / Edit Fields** — преобразование полей
- **IF / Switch** — ветвление логики
- **Merge** — склейка двух веток
- **Telegram → Send Message** — отправка сообщения ботом
- **Postgres / Supabase** — работа с БД

## Клавиши холста

| Действие | Шорткат |
|---|---|
| Двигать весь холст | средняя кнопка мыши ИЛИ `Space + drag` |
| Zoom | `Ctrl + scroll` |
| Вернуться к центру | кнопка-«щётка» в правом нижнем углу |
| Активировать ноду вручную | `▶` на самой ноде |
| Запустить весь workflow | `Execute Workflow` (правый верхний угол) |
| Удалить ноду без разрыва связей | `Deactivate` (временно выключить) |

> Из урока 4: «Деактивировать ноду — это очень важная кнопочка. Если нода не нужна прямо сейчас, но мы не хотим разрывать цепочку — мы её деактивируем. Сквозь неё проходят данные как через пустое место».

## Свойства workflow (Settings)

Правый верхний угол → три точки → **Settings** (появляется только после того, как в workflow есть хотя бы одна нода и он сохранён).

| Параметр | Рекомендация | Зачем |
|---|---|---|
| Execution Order | **v1 (recommended)** | последовательность нод: слева направо, сверху вниз |
| Error Workflow | подключить свой `error-handler` workflow | узнать, что что-то сломалось |
| Timezone | Europe/Moscow (если календарь/время важно) | иначе даты «поплывут» |
| Save Execution Success | **Don't Save** | не забивать диск |
| Save Execution Error | **Save** | иначе не увидишь ошибки |
| Save Manual Executions | Don't Save (на проде) / Save (при разработке) | та же логика |
| Timeout Workflow | по желанию | ограничить время выполнения |
| Time Saved per Execution | 180 (если процесс экономит 3 ч/день) | маркетинговая метрика |

> Из урока 5: «Катастрофическое переполнение жёсткого диска... основная трата — каждое выполнение записывается. Управляем этим: save error / don't save success».

## Анти-паттерны (не делай так)

- ❌ Цитировать огромный промпт в каждом сообщении (вынеси в Telegram Bot Menu / переменную).
- ❌ Inline-кнопка с `callback_data` > 64 байт (Telegram молча обрежет) → см. [ENVIRONMENT.md §7.2].
- ❌ Использовать webhook на localhost → см. [10-common-errors.md §10.4](10-common-errors.md).
- ❌ Хранить API-ключ в ноде руками → всегда через Credentials.
- ❌ Один мега-workflow на 200 нод → разбивай на саб-workflow через Execute Workflow.
- ❌ Скриншот ошибки → текст ошибки: «Пришлите текстом, я по коду найду решение» (формула George Erman).

## Золотое правило

> Из чата, Александр: «Сетевой код — не угадывай, а смотри в ноду и в логи». Если ИИ-помощник не помогает, открой «Executions» → красная нода → копируй текст ошибки и гугли / спрашивай.
