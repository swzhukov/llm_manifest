# 03 — Настройки workflow (Settings)

> Основано на уроке 5 курса George Erman + обсуждения в чате.

Settings открываются через **три точки** в правом верхнем углу → `Settings`.

⚠️ **Важно:** пункт `Settings` появляется **только после того, как в workflow есть хотя бы одна нода и workflow сохранён**. Это частая причина «не могу найти настройки» у новичков.

## 3.1 Execution Order

| Значение | Что делает |
|---|---|
| **v1 (recommended)** | ноды выполняются слева направо, сверху вниз; параллельные ветки — последовательно |
| v0 (legacy) | старая логика, не рекомендуется |

✅ **Best practice:** всегда оставляй **v1**. Это правильная последовательность, проверено.

## 3.2 Error Workflow

Сюда подставляешь **другой workflow**, который должен сработать, если в текущем произошла ошибка.

Паттерн:
1. Создаёшь workflow `error-handler` (или `error-router`).
2. В нём: **Error Trigger** → IF (есть `execution.id`) → Telegram-сообщение админу «упало: ...».
3. В каждом продакшн-workflow: Settings → Error Workflow → выбираешь `error-handler`.

⚠️ **Из чата:** «Триггер error не даёт инфу об id юзера, у которого ошибка. AI предлагает 2 варианта: 1) настройка в критических нодах (их слишком много) 2) через Supabase сохранять id, потом в error-workflow взять их и отправить в нужный чат». → Решение: храни `chat_id` в БД, тяни его в error-workflow.

Подробнее — [04-error-handling.md](04-error-handling.md).

## 3.3 Timezone

Для 95% задач неважно. Но если в workflow есть:
- Google Calendar
- Cron, привязанный к «9 утра»
- `Date & Time` ноды
- Сравнение дат

...то **обязательно** выстави свою тайм-зону. По умолчанию — UTC, и «9 утра в Москве» будет 6 утра.

✅ **Best practice:** для РФ-проектов → `Europe/Moscow`.

## 3.4 Save Execution Success / Error / Manual

Это **главная настройка против переполнения диска**.

| Что сохраняем | Продакшн | Разработка |
|---|---|---|
| **Error** | **Save** (иначе не увидишь ошибки) | Save |
| **Success** | **Don't Save** | Save (для отладки) |
| **Manual (запуски по кнопке)** | **Don't Save** | Save |

> Из урока 5: «Катастрофическое переполнение жёсткого диска... основная трата — каждое выполнение записывается. Управляем этим: сейв error / don't save success».

### Что выставить, чтобы диск не забился за неделю

```
Save Error Success:        Save
Save Execution Success:    Don't Save
Save Manual Executions:    Don't Save
```

Дополнительно: можно периодически чистить старые executions:

```bash
# В n8n UI: Settings → Usage → Clean historical executions
# Или через API: DELETE /api/v1/executions?ids=...
```

## 3.5 Timeout Workflow

Ставит максимальное время выполнения. Если workflow «висит» (например, ждёт ответ от API, который завис) — через N секунд прервётся.

✅ **Best practice:** для большинства задач **не нужен**. Ставь, только если:
- Workflow реально может зависнуть (HTTP Request без retry-логики).
- Есть лимиты по CPU / памяти (долгий workflow = жрёт ресурсы).

Из урока 5: «Можно использовать, но я лично не использую».

## 3.6 Time Saved per Execution

Маркетинговая метрика. Если workflow экономит клиенту N минут в день, ставишь N, и в дашборде видишь «сколько часов сэкономлено за всё время».

Пример: автоматизация отвечает на письма вместо тебя 3 часа в день → ставишь **180 минут**.

Из урока 5: «Когда вы нажмёте "Сохранить" и перейдёте в ваш дашборд, у вас в Time Saved вы будете видеть, сколько в реальности ваши workflow сохранили вам рабочего времени».

## 3.7 Caller Policy (для Execute Workflow sub-workflow)

Если используешь **Execute Workflow** (вызов sub-workflow), то в настройках **sub-workflow** есть Caller Policy:

| Значение | Кто может вызвать |
|---|---|
| Any | любой workflow |
| Workflows in same project | только из того же проекта |
| Specific workflows | явно указанные |

✅ **Best practice:** для sub-workflow внутри своего проекта → `Workflows in same project` (безопаснее, чем `Any`).
