# 10 — Common Errors (типичные ошибки)

> Собрано из всех 10 тредов чата. Это «живая» боль, которая стоит времени — фиксируй, чтобы не наступать повторно.

## 10.1 «Lost connection to server» / 500

**Симптом:** workflow запускается, в UI ошибка 500, иногда «lost connection».

**Причины:**
1. **Долгий workflow превысил таймаут UI** — workflow всё ещё работает на сервере, просто UI отвалился. **Проверь Executions** — там будет результат.
2. **OOM на сервере** — RAM кончилась. 1 ГБ — минимум, 2 ГБ комфортно.
3. **Слишком большой файл в webhook** — увеличь `N8N_PAYLOAD_SIZE_MAX`.

**Решение:** подождать, проверить Executions, если воспроизводится — оптимизировать workflow.

## 10.2 Возвращается HTML вместо JSON (санкции)

**Симптом:** HTTP Request к внешнему API возвращает HTML-страницу, а не данные.

**Причина:** IP-адрес сервера попал в санкционный список (для западных API) или сайт требует авторизацию.

**Решение (из чата, декабрь 2025):**
> «Возвращает HTML... ip сервера попадает под санкционную страну, у меня Финляндия»

1. **Сменить локацию сервера** — российский / казахский дата-центр.
2. **Через прокси/VPN** — поднять на сервере, например, `3proxy`.
3. **Использовать российский аналог API** — если есть.

## 10.3 Обновление стёрло .env

**Симптом (январь 2026):**
> «После обновления до 2.1.5 у меня в .env всё пропало само, но бэкап credentials начал работать».

**Решение:**
1. **Бэкапь .env ДО обновления:** `cp .env .env.backup-$(date +%F)`
2. **Бэкапь Postgres:** `docker exec n8n-postgres pg_dump -U n8n n8n > backup.sql`
3. **Используй только upgrade-скрипт**, не install (см. [01-installation.md §1.4](01-installation.md)).
4. **После обновления проверь:** n8n запустился, credentials на месте, workflows активны.

## 10.4 Webhook недоступен извне

**Симптом:** Telegram-бот не отвечает, хотя локально работает.

**Причина:** webhook указывает на `http://localhost:5678/...` или на IP, недоступный из интернета.

**Решение:**
- **Прод:** webhook через публичный домен + Caddy (он уже в docker-стеке). URL будет `https://n8n.yourdomain.ru/webhook/...`.
- **Dev:** используй community-ноду `@mentoster/n8n-nodes-telegram-polling` (long-polling, без webhook).
- **Туннель:** `ngrok http 5678` — даёт временный публичный URL.

## 10.5 «AI service is receiving too many requests from you»

**Симптом:** AI Agent выдаёт ошибку «too many requests».

**Причина:** rate-limiter на стороне провайдера (OpenRouter, OpenAI при бесплатном tier).

**Из чата (март 2026):**
> «Отправляла только один запрос и мне сразу же приходило "слишком много запросов"... через 2–5 минут, и даже на следующий день система отвечала, что много запросов».

**Решение:**
1. **Подождать 2–5 минут** (обычно этого достаточно).
2. **Переключиться на платный tier** — лимиты выше.
3. **Использовать другой роутер** — OpenRouter, Anthropic, YandexGPT.
4. **Снизить частоту** — добавить `Wait` ноду между запросами.

## 10.6 YandexGPT не подключается

**Симптом:** нода Chat Model показывает пустой список моделей или ошибку авторизации.

**Причины и решения:**

| Причина | Решение |
|---|---|
| Перепутал API Key и Account ID | в credentials нужен **API Key** (секрет, не ID) |
| Облако не привязано к Billing | Cloud Console → Billing → привязать облако |
| Нет ролей у сервисного аккаунта | добавить `ai.foundationModels.user` и др. |
| Неправильный Base URL | `https://llm.api.cloud.yandex.net/v1` |

## 10.7 SQL не выполняется в Supabase UI

**Симптом:** SQL в Supabase → SQL Editor возвращает ошибку.

**Из чата (март 2026):**
> «В супу зайти можете? В окно где вставляли запрос — там вставляете любой SQL. Это если в супе, если через n8n, то через ноду Postgres».

**Решение:** если UI не срабатывает — выполни SQL через **Postgres-ноду** в n8n:
```
Postgres (Execute Query)
  Query: "CREATE EXTENSION IF NOT EXISTS vector; CREATE TABLE ...;"
```

## 10.8 Telegram: «Chat not found»

**Симптом:** Send Message возвращает 400 с «chat not found».

**Причины:**
1. **chat_id — строка вместо числа** (или наоборот). Telegram-нода требует **строку** или **число**.
2. **Юзер не нажал /start** — бот не может писать первым.
3. **chat_id в неправильном месте** — у Telegram Trigger это `$json.message.chat.id`, у callback — `$json.callback_query.message.chat.id`.
4. **Группа, а не пользователь** — нужен chat_id группы (отрицательное число).

**Диагностика:**
```
HTTP Request
  method: GET
  url: https://api.telegram.org/bot{{$env.TOKEN}}/getChat
  qs: { chat_id: "{{$json.chat_id}}" }
```

Вернёт информацию о чате или ошибку с описанием.

## 10.9 Бот не видит сообщения в группе

**Симптом:** бот в групповом чате, но не реагирует на сообщения.

**Причина (из чата, январь 2026):**
> «Telegram Copilot не работает автоматизация... поступило сообщение в чат, а не мне лично, я его должен обработать».

**Решение:**
1. **Bot API** (стандарт) — бот видит только:
   - `/команды` (если упомянуть бота через `@botname`)
   - Сообщения, на которые ему ответили (`reply_to_message`)
   - Сообщения с упоминанием `@botname`
2. **User-bot (Telethon / Pyrogram)** — нужен **отдельный Python-скрипт**, не n8n-нода. Бот авторизуется как пользователь.
3. **Получить все сообщения группы** — для этого в BotFather включить `Privacy mode = DISABLED` (через `/setprivacy` → выбрать бота → Disable).

## 10.10 Media Group: «files must have .jpg extension»

**Симптом:** отправка альбома фото в Telegram падает.

**Из чата (октябрь 2025):**
> «Хочу чтоб в телеграм фото приходило в виде галереи, а не раскиданно... ИИ говорит, что есть Telegram-узел Media Group, я собрал бинарные файлы в кучу, но он постоянно выдает ошибку».

**Решение:** используй **HTTP Request** к `sendMediaGroup` вместо встроенной ноды. Подробнее: [07-telegram-bots.md §7.8](07-telegram-bots.md).

## 10.11 Бесплатная версия n8n: «не работает 24/7»

**Симптом (март 2026):**
> «На бесплатной версии агент не будет работать 24/7?»
> «Нет кнопки активации»

**Ответ (Анастасия):**
> «Кнопка Public». (Имеется в виду — Publish / Activate workflow, не Public Link).

**Решение:**
- **Активация через тумблер** (зелёный переключатель) — это и есть «24/7» на бесплатной версии.
- **Publish** — это для **опубликованной версии** (отдельная фича для шаринга).
- **Public** — это **поделиться ссылкой** (read-only).

Если n8n на своём сервере — активация работает без ограничений. Ограничения только у n8n.cloud trial.

## 10.12 Бесконечный цикл / лимит execution

**Симптом:** workflow запускается бесконечно, диск забит executions.

**Решение:**
1. **Settings → Error/Save Don't Save** — см. [03-workflow-settings.md §3.4](03-workflow-settings.md).
2. **Проверь условие выхода** — IF-каскад должен явно завершать workflow.
3. **Stop and Error** — при невозможности продолжить.

## 10.13 Drag-and-drop кнопки: «callback_data пустой»

**Симптом:** inline-кнопка нажата, но в workflow приходит пустой callback.

**Причина:** `callback_data > 64 байт` (Telegram обрезает).

**Решение:** маппинг через Supabase (короткий код → длинное значение). Подробнее: [07-telegram-bots.md §7.2](07-telegram-bots.md).

## 10.14 Не отрабатывает триггер в Telegram

**Симптом:** сообщение отправлено боту, но workflow не запускается.

**Диагностика:**
1. **Workflow активен?** Зелёный тумблер → должен быть зелёным.
2. **Credentials правильные?** Telegram API → правильный токен.
3. **Webhook зарегистрирован?** Telegram Trigger автоматически регистрирует webhook при активации. Если не сработало — смотри Executions.
4. **Webhook URL правильный?** Должен быть публичным HTTPS.

## 10.15 Cron Schedule не срабатывает

**Симптом:** Schedule Trigger по расписанию не запускается.

**Причины:**
- **Workflow не активирован** — поставь тумблер в зелёное.
- **Cron-выражение неверное** — проверь через crontab.guru.
- **Часовой пояс** — учти, по умолчанию UTC (см. [03-workflow-settings.md §3.3](03-workflow-settings.md)).

## 10.16 Большое видео/аудио не доходит

**Симптом:** Telegram Trigger не «ловит» большой файл.

**Решение:**
- **n8n PAYLOAD SIZE:** увеличить `N8N_PAYLOAD_SIZE_MAX=100mb` в .env.
- **Telegram Bot API лимит:** 20 МБ на скачивание через `getFile`. Для большего — использовать собственный Telegram Client API.
- **Save data location** — настройки `N8N_BINARY_DATA_MODE=filesystem` (по умолчанию) или S3.

## 10.17 Bot Menu не отображается

**Симптом:** `/start` показывает стандартное сообщение, не кастомное меню.

**Решение:**
1. `setMyCommands` — настройка **один раз** при старте (см. [07-telegram-bots.md §7.9](07-telegram-bots.md)).
2. Trigger на `/start` в workflow → отправь приветствие + кнопки.
3. Telegram **кеширует** меню — после изменения подождать 5–10 минут.

## 10.18 Код в Code-ноде падает

**Симптом:** `ReferenceError`, `TypeError` в Code-ноде.

**Решение:**
1. **Используй `$input.all()`** для доступа ко всем items.
2. **`$input.first()`** для первого.
3. **Возвращай массив** объектов (даже если один).
4. **Логируй:** `console.log(JSON.stringify($json))` в начале кода.
5. **Используй Try/Catch** для устойчивости.

## 10.19 «Текст ошибки слишком длинный» в Telegram-сообщении

**Симптом:** error handler отправляет 50-килобайтную ошибку, Telegram возвращает 400.

**Решение:**
```
Code
  ↓
if (error.message.length > 4000) {
  error.message = error.message.substring(0, 4000) + '...[truncated]';
}
return { json: { error: error.message } };
```

## 10.20 Скриншот вместо текста ошибки

**Из чата (неоднократно):**
> «Скиньте мне в личку текст ошибки»

⚠️ **Скриншот ошибки — антипаттерн.** По картинке трудно гуглить, невозможно копировать, она теряет качество. **Всегда копируй текст ошибки** и присылай текстом.

Хорошая практика:
```
Error node: HTTP Request
Error message: 401 Unauthorized - "API key invalid"
Trace: at /workflow/telegram-bot/exec/1234
Execution ID: abc-123-def
```

## 10.21 Нет звука / не приходит voice

**Симптом:** бот отправляет .mp3 файл, но Telegram показывает его как документ, а не как голосовое.

**Решение:**
- Использовать `sendVoice` API вместо `sendAudio`.
- Метод `sendVoice` принимает **OGG** (Opus) — конвертировать через ffmpeg:
  ```bash
  ffmpeg -i input.mp3 -c:a libopus -b:a 32k output.ogg
  ```
- Или отправлять через Telegram-ноду **Send Voice** (не Send Audio).
